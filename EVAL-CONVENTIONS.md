# Plugin Eval Conventions

Conventions for plugin skill evaluations. This PR (evals/plugin-skills) introduced
eval workspaces for two plugins that evolved separately. This doc captures the
canonical patterns so future plugins converge rather than diverge.

## Eval Definitions (`evals/evals.json`)

Each skill has its own eval file at `<plugin>/skills/<skill-name>/evals/evals.json`.

```json
{
  "skill_name": "skill-name",
  "evals": [
    {
      "id": 1,
      "name": "descriptive-eval-name",
      "prompt": "Full user prompt text — never truncated",
      "expected_output": "What success looks like",
      "files": ["evals/files/input.xlsx"],
      "expectations": [
        "Each expectation explains what to check AND why it matters"
      ]
    }
  ]
}
```

Required fields: `id`, `name`, `prompt`, `expected_output`, `files`, `expectations`.

The `name` field is used by the skill-creator agent to name workspace directories.
Omitting it causes eval directories to fall back to generic `eval-0`, `eval-1` names.

## Eval Workspaces

Workspaces hold iteration artifacts (outputs, grading, benchmarks). They are
generated outputs, not source — treat them as evidence that evals ran.

### Layout: per-skill workspaces

```
<plugin>/
├── skills/
│   ├── skill-a/
│   └── skill-b/
├── skill-a-workspace/
│   ├── iteration-1/
│   │   ├── benchmark.json
│   │   ├── eval-name-1/
│   │   │   ├── eval_metadata.json
│   │   │   ├── with_skill/
│   │   │   │   ├── grading.json
│   │   │   │   ├── timing.json
│   │   │   │   └── outputs/
│   │   │   └── without_skill/
│   │   │       ├── grading.json
│   │   │       └── outputs/
│   │   └── eval-name-2/
│   └── iteration-2/
└── skill-b-workspace/
```

Per-skill workspaces are the default. Each skill's eval history is self-contained
and can be re-run independently.

A shared workspace (`evals-workspace/` containing evals for multiple skills) is
acceptable when a plugin has many lightweight skills that were evaluated together
in a single session. Both patterns work with the tooling.

### `eval_metadata.json`

```json
{
  "eval_id": 1,
  "eval_name": "descriptive-name",
  "prompt": "Full prompt text — never truncated",
  "assertions": [
    "Assertion text matching what the grader evaluates"
  ]
}
```

Use `assertions` for the criteria field (not `expectations`, not both).
Always include the full prompt — truncated prompts break re-runs.

### `grading.json`

The viewer and aggregation scripts depend on these exact field names:

```json
{
  "expectations": [
    {
      "text": "The assertion being evaluated",
      "passed": true,
      "evidence": "Why it passed or failed"
    }
  ],
  "summary": {
    "passed": 5,
    "failed": 1,
    "total": 6,
    "pass_rate": 0.83
  }
}
```

Required fields in each expectation: `text`, `passed`, `evidence`.
Do NOT use `name`/`met`/`details` — the viewer will render empty cells.

Optional fields (`claims`, `eval_feedback`, `user_notes_summary`, `execution_metrics`)
are passed through by the aggregation script but not rendered. Include them if the
grader produces them; don't require them.

### `timing.json`

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

Captured at subagent completion — cannot be backfilled. Include when available.
The aggregation script uses `grading.json` timing first, falls back to `timing.json`.

### `benchmark.json`

Generated by `aggregate_benchmark.py` from grading data. Do not hand-author.
The `configuration` field must be exactly `"with_skill"` or `"without_skill"` —
the viewer uses these strings for color coding.

## Baselines

Iteration 1 must include both `with_skill/` and `without_skill/` runs.
Later iterations may drop baselines when iterating on skill improvements only.

## Trigger Eval Sets

After the skill is stable, create `<plugin>/skills/<skill-name>/trigger_eval_set.json`
with 20 queries (mix of should-trigger and should-not-trigger) for description
optimization via `scripts/run_loop.py`.

## Known Divergences in This PR

The two plugins in this PR were developed in parallel and have these differences.
Both work with the tooling. Future plugins should follow the conventions above.

| Aspect | idfa-financial-architect | sales-revops-marketing |
|---|---|---|
| Workspace layout | Per-skill | Shared (`evals-workspace/`) |
| `evals.json` `name` field | Present | Missing |
| `eval_metadata.json` criteria field | `assertions` | `assertions` + `expectations` (duplicated in iter-2) |
| `eval_metadata.json` prompts | Full text | Truncated in iter-1, full in iter-2+ |
| `grading.json` schema | Lean | Rich in iter-1 (claims, feedback), lean in iter-2+ |
| `timing.json` | Present in iter-2+ | Not captured |
| Trigger eval sets | Present | Not created |
| Description optimization | Run for financial-architect | Not run |
