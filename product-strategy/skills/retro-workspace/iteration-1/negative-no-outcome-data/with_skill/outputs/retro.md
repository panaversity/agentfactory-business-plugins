## Retro Request: Search Feature — Too Early

I can't run a meaningful retrospective on the search feature right now.

**Why not**: The search feature shipped last week. A product retrospective requires
4-12 weeks of outcome data — adoption rates, support ticket trends, NPS movement,
and business metric impact. At one week post-launch, the data is too thin to draw
conclusions. Running a retro without outcome data produces opinions and blame, not
evidence-based learning.

**Recommended timeline**: Schedule the retro for 6-8 weeks post-launch. By then you'll
have enough data to evaluate whether the feature solved the problem it was designed to
solve.

**What you CAN do now**: If you want to review how the delivery went while it's fresh,
I can run a delivery-focused review covering Q2 (Did we build it as intended?):

- Was it delivered on time?
- Were there scope changes during development?
- Did engineering flag any spec quality issues?
- Were all acceptance criteria verified?

This captures the process learnings while they're fresh, and you can fold them into
the full retro once outcome data is available.

**To proceed with a delivery review**, provide:
- Planned vs actual delivery date
- Any scope changes during development
- Any spec ambiguity issues flagged by engineering
- AC verification status

**To schedule the full retro**, set a calendar reminder for ~6-8 weeks post-launch
with a data collection checklist:
- [ ] Adoption: % of target users who have used search at least once
- [ ] Engagement: search frequency and session depth
- [ ] Support impact: search-related tickets vs. pre-launch baseline
- [ ] Business metric: primary success metric from the spec
